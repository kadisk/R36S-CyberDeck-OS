# Resultado — Fase 5 (web via X11) — ✅ UI WEB NA TELA DO R36S

**Data:** 2026-06-15
**Resultado:** 🎉 **SUCESSO** — a CyberDeck UI (HTML/CSS/JS) renderiza no R36S físico,
em **Chromium kiosk** sobre **Xorg (fbdev)**, no **kernel BSP** (painel funciona).

## A solução que venceu

```
U-Boot + kernel 4.4 BSP + DTB rk3326-r35s-linux.dtb (clone do boot ArkOS) → PAINEL ACENDE
   ↓ root=UUID
Rootfs Debian arm64 (bookworm)
   ↓
Xorg (driver fbdev em /dev/fb0, render por software)
   ↓
Chromium --kiosk --app=file:///usr/share/cyberdeck-ui/public/index.html
```

Gerado por `scripts/build-x11-rootfs.sh` (registrado como imagem `x11`).

## Por que ESTE caminho funcionou (e os outros não)

| Caminho | Bloqueio |
|---------|----------|
| BSP + cog/cage (Wayland) | GBM do **blob Mali** antigo (sem `gbm_bo_get_offset`) — Fase 4 |
| mainline + Panfrost + cog | **painel** não sobe no mainline (nem no Arch-R) — Fase 5a |
| distros prontas (rocknix/armbian/multipanel) | só o **kernel+DTB BSP do ArkOS** dirige este painel |
| **BSP + Xorg(fbdev) + Chromium** | ✅ **nenhum** — usa o painel que funciona + X11 (sem GBM) |

**Lição central:** neste R36S, o painel (`elida,kd35t133`) só é dirigido pelo
**kernel+DTB BSP**. E o web acelerado por Wayland/Mali está bloqueado pelo blob
antigo. A combinação que entrega web é **BSP (painel) + X11 (sem GBM) + navegador
por software** (a UI é leve, então software rendering basta).
